# EngliAI 项目优化方案

---

## 概述

本文档基于对项目代码、业务逻辑、数据库结构及前端页面的全面审查，从**代码实现逻辑**、**业务逻辑与流程**、**前端页面显示**三个维度系统梳理可优化点。每个优化项标注优先级：**P0（紧急）> P1（高）> P2（中）> P3（低）**。

---

## 一、代码实现逻辑优化

### 1.1 消除重复代码

#### P1 | 重复的 `parseLong()` — 16+ 个 Servlet 中拷贝
**文件**: `Servlet/AdminArticleCreateServlet.java`, `AdminArticleDeleteServlet.java`, `AdminArticleGetServlet.java`, `AdminArticleListServlet.java`, `AdminArticleUpdateServlet.java`, `AdminKickUserServlet.java`, `AdminOnlineUsersServlet.java`, `AdminRequestLogsServlet.java`, `AdminUserCreateServlet.java`, `AdminUserDeleteServlet.java`, `AdminUserListServlet.java`, `AdminUserResetPwdServlet.java`, `CefrProgressServlet.java`, `UserUpdateStudyPurposeServlet.java`, `VipExchangeServlet.java`, `VocabTestSubmitServlet.java` 等

每个文件都有完全相同的 `parseLong(String s)` 方法。**建议**: 移入 `Utils.ServletUtils` 公共基类或工具方法。

#### P1 | 重复的 `escapeJson()` — 7+ 处各自实现，部分不完整
**文件**: `AdminArticleGetServlet.java`, `AITitleServlet.java`, `ArticleDetailServlet.java`, `ArticleDiagnosticServlet.java`, `ArticleListServlet.java`, `AIService.java` 内部类, `WordService.java` 内部类

`JsonUtil.escapeJson()` 已有完整实现（处理 `\b`, `\f`, `\n`, `\r`, `\t`, `\\`, `\"`)，但多处复用不完整版本。如 `ArticleDiagnosticServlet` 只转义 `\` 和 `"`，缺少 `\n`/`\r`/`\t` 转义。**建议**: 统一使用 `JsonUtil.escapeJson()` 或 `JsonUtil.strVal()`，删除所有本地副本。

#### P1 | 重复的管理员认证模式 — 12 个 Servlet 中拷贝
```java
Long adminUserId = parseLong(request.getParameter("adminUserId"));
if (!adminService.isAdmin(adminUserId)) {
    response.getWriter().write(JsonUtil.error("无管理员权限"));
    return;
}
```
**建议**: 封装为 `ServletUtils.authenticateAdmin(request, adminService)` 方法。

#### P1 | 手写 JSON 字符串拼接 — 40+ 文件
使用 `StringBuilder` / 字符串拼接构建 JSON 响应，大量重复、易出错，且对用户生成内容（如文章标题）无注入保护。**建议**: 尽管项目当前避免第三方依赖，但可构建一个轻量级的 `JsonBuilder` 工具类来统一序列化；或引入 `org.json` / `gson` 等轻量库。

#### P2 | 多个 Servlet 绕过 Service/DAO 层直接操作数据库
**文件**: `ArticleDetailServlet.java`, `ArticleListServlet.java`, `ArticleDiagnosticServlet.java`

直接在 Servlet 中调用 `DBUtil.getConnection()` 并拼接 SQL，破坏分层架构。**建议**: 通过 `ArticleService` → `ArticleDAO` 层访问。

#### P2 | `AIService.java` 过于庞大（1315行）
包含单词例句/查词/翻译/文化分析/出题/评估/标题生成/JSON 解析/JSON 转义等，8 个内部类。**建议**: 拆分为 `AIClient`（API通信）、`AIWordService`、`AIQuizService`、`AIAssessmentService`。

#### P2 | `UserDAOImpl.java` 中 SELECT 列名列表重复多次
**文件**: `UserDAOImpl.java` — 14 列的 SELECT 列表在 `findByUsername`, `findById`, `findAll` 中逐字重复。**建议**: 定义为 `private static final String USER_COLUMNS` 常量。

#### P2 | 分散的 JSON 手动解析 — 6+ 个 Servlet 用 `indexOf` 提取 JSON 字段
**文件**: `ArticleQuizServlet.java`, `CulturalNotesServlet.java`, `ParagraphTranslationServlet.java`, `AssessmentEvaluateServlet.java`, `AssessmentGenerateServlet.java`, `VocabTestCardResultServlet.java`, `VocabTestOptionServlet.java`

使用 `responseContent.indexOf("\"title\":\"")` 等手法手工提取 JSON 字段，脆弱且无法处理嵌套。**建议**: 引入标准 JSON 解析库。

#### P3 | Vue 脚手架残留文件（8个无用文件）
**文件**: `HelloWorld.vue`, `TheWelcome.vue`, `WelcomeItem.vue`, `icons/IconCommunity.vue`, `icons/IconDocumentation.vue`, `icons/IconEcosystem.vue`, `icons/IconSupport.vue`, `icons/IconTooling.vue`

项目使用 `@iconify/vue`，这些 Vue CLI 脚手架模板文件未被任何页面引用。**建议**: 删除，减少打包体积。

### 1.2 安全加固

#### P0 | 管理员操作绕过会话认证 — 关键安全漏洞
**文件**: `Filter/LoginFilter.java` 第 54 行

`/api/admin/` 被列入公开路径前缀，所有管理端点的"认证"仅依赖客户端传入的 `adminUserId` 请求参数。任何知道 admin 用户 ID 的人都可以伪装管理员操作。

**建议**: 
1. 从 `PUBLIC_API_PREFIXES` 中移除 `/api/admin/`
2. 管理员身份从 `session.getAttribute("role")` 获取，而非请求参数
3. 在 Filter 或 Servlet 层面校验会话中的 role == "admin"

#### P0 | MD5 + 盐 存储密码 — 严重安全隐患
**文件**: `Utils/PasswordUtil.java` 第 26-38 行

- MD5 已被破解，GPU 可每秒计算数十亿次
- 无迭代拉伸（bcrypt/scrypt/PBKDF2 默认数千次迭代）
- `verify()` 使用 `String.equals()`，非恒定时间比较，存在时序侧信道

**建议**: 替换为 bcrypt（`org.mindrot:jbcrypt`），建议 work factor ≥ 12。迁移策略：用户登录时，若旧 MD5 验证通过，用 bcrypt 重新哈希并更新存储。

#### P0 | 生产凭证泄露在版本控制中
**文件**: `config.properties` 及 Git 历史

API 密钥 (`sk-0e2bd...`) 和数据库密码 (`mCeeTPH...`) 已在 Git 仓库中暴露。**建议**:
1. 立即在 DeepSeek 后台吊销旧 API Key，生成新 Key
2. 更换数据库密码
3. 确认 `config.properties` 已在 `.gitignore` 中（已完成）
4. 用 `git filter-branch` 或 `BFG Repo-Cleaner` 清理历史记录中的敏感信息

#### P1 | 部分用户写端点列为公开
**文件**: `Filter/LoginFilter.java` 第 38-41 行

`/api/user/experience`, `/api/user/vip-exchange`, `/api/user/cefr-progress`, `/api/user/update-study-purpose` 均列在 `PUBLIC_API_PATHS` 中，且从请求参数读取 `userId` 而非会话。任何用户可为其他用户操作。

**建议**: 移出公开列表，改为从 `session.getAttribute("userId")` 读取操作对象。

#### P2 | 密码盐字符集仅为字母
**文件**: `Utils/PasswordUtil.java` 第 8 行 — `CHAR_POOL = "a-z A-Z"`（仅52字符）

**建议**: 扩展为 `A-Za-z0-9!@#$%^&*` 以增加熵。

#### P2 | `sendUnauthorized()` 中消息未转义
**文件**: `Filter/LoginFilter.java` 第 166 行 — 直接将字符串拼入 JSON

**建议**: 使用 `JsonUtil.escapeJson(message)`。

### 1.3 数据库连接与性能

#### P0 | 无连接池
**文件**: `Utils/DBUtil.java` 第 21 行

每次调用 `DriverManager.getConnection()` 新建 TCP 连接。`WordDAOImpl.findByWordInAllTables()` 一次请求连开 6 个连接。并发下会迅速耗尽 MySQL 的 `max_connections`。

**建议**: 引入 HikariCP（`com.zaxxer:HikariCP:5.1.0`），配置最小/最大连接池大小。

#### P1 | JDBC URL 缺少超时参数
**文件**: `config.properties` 第 12 行

缺 `connectTimeout` 和 `socketTimeout`，MySQL 挂起时应用线程无限阻塞。

**建议**: 添加 `connectTimeout=5000&socketTimeout=30000`。

#### P2 | 懒初始化同步列表的竞态条件
**文件**: `Listener/RequestLogListener.java` 第 79-86 行 — `getLogList()` 非线程安全

两请求同时触发初始化时，后一个会覆盖前一个的列表对象。**建议**: 在 `AppContextListener` 中预初始化。

#### P3 | `DBConnectTest.java` 在主源码中存在
**文件**: `Utils/DBConnectTest.java`

非测试，是 DDL 工具。**建议**: 移入 `src/test/java/` 或移除。

### 1.4 异常处理与日志

#### P1 | 不统一的错误处理策略

| 层次 | 策略 | 示例 |
|------|------|------|
| DAO | 全部包装为 `RuntimeException` | `throw new RuntimeException("查询失败", e)` |
| Service | 返回 null / 错误字符串 / -1 / -2 | `UserService.checkin()` 返回 -1/-2 |
| Servlet | 零星 try-catch 写 JSON 错误 | 不一致 |

**建议**: 约定统一的返回格式（如 `Result<T>` 包装类），或定义 `DataAccessException`、`EntityNotFoundException` 等自定义异常，使用全局异常处理 Filter 转换为 JSON 响应。

#### P1 | `System.out.println` / `System.err.println` 遍布全项目（60+ 处）
**建议**: 引入 SLF4J + Logback，支持日志级别控制和格式化输出。

#### P2 | 多个空的 catch 块
- `MonitorService.getOnlineUsers()` 第 90 行: `catch (Exception ignored) {}`
- `AdminService.logAction()` 第 363 行: 日志写入失败静默吞掉

**建议**: 至少记录警告日志。

#### P3 | `web.xml` 缺少全局错误页
无 `error-page` 声明，未捕获异常会暴露栈追踪给用户。**建议**: 配置 404/500 错误页，JSON API 异常由全局处理 Filter 统一响应。

## 二、业务逻辑与流程优化

### 2.1 数据库结构

#### P0 | `last_checkin` 字段语义过载 — 数据完整性 Bug
**文件**: `DAO/UserDAOImpl.java` 第 333-340 行, `Service/UserService.java` 第 155-183 行

`last_checkin` 字段同时用于：(1) 存储每日签到时间，(2) 存储 VIP 到期时间。当 VIP 用户签到后，`updateCheckin()` 会覆盖 VIP 到期时间到 `last_checkin`，导致该用户不受限使用 VIP 功能。

**建议**: 新增 `vip_expire_at DATETIME` 列，解除语义过载。

#### P1 | 六张独立词库表导致代码膨胀
**表**: `junior`, `senior`, `cet_4`, `cet_6`, `postgraduate`, `toefl`（结构完全相同）

导致 6 个实体类、6 次 `findByWordInAllTables()` 的循环查询、Schema 变更需改 6 张表。"考研"对应表名 `postgraduate`/类名 `Graduate`/ENUM `考研`，三处名称不一致。

**建议**: 统一为单张 `word` 表，增加 `stage ENUM` 字段区分阶段，索引 `(word, stage)`，1 次 SQL 替代 6 次循环。

#### P1 | `profile` 字段过载
**文件**: `DAO/UserDAOImpl.java` 第 155-183 行 — 手动 `indexOf` 解析 JSON `profile` 字段

```
profile 字段同时存储: VIP状态 / CEFR进度 / 混合状态
读取方式: indexOf("{") / indexOf("\"cefrProgress\"") 手写扫描
```

**建议**: 新增 `cefr_progress INT` 和 `vip_expire_at DATETIME` 列，废弃 profile JSON hack。

#### P2 | `admin_action_log` 无主键
**文件**: 数据库表 `admin_action_log` — 仅有两个复合索引，InnoDB 创建隐藏 row_id

**建议**: 添加 `id BIGINT AUTO_INCREMENT PRIMARY KEY`。

#### P2 | `vocab_test` 唯一键与文档矛盾
**文件**: `数据库结构说明.md` 第 96-98 行 vs 第 105 行 — UNIQUE KEY uk_word 与"同一单词可能出现多次"矛盾

**建议**: 确认设计意图并统一。

#### P3 | 无外键约束 — 级联删除由应用层实现
**建议**: 对 `user_learned_article`、投票表等加 `ON DELETE CASCADE`，减少应用层手动清理代码。

### 2.2 业务逻辑

#### P0 | 有效学习阶段定义不一致
**文件**: `AdminService.java` vs `UserService.java`

| 文件 | 有效阶段 | 个数 |
|------|---------|------|
| `AdminService.VALID_STAGES` | 初中/高中/四级/六级/考研/托福/期刊/原著/网络新闻 | 9 |
| `UserService` validStages | 初中/高中/四级/六级/考研/托福 | 6 |

管理员可创建"期刊/原著/网络新闻"阶段的用户，但用户自己无法注册这些阶段。`WordService.STAGE_EXPECTED_VOCAB` 也只有 6 个阶段。

**建议**: 将有效阶段提取到 `ConfigUtil` 中的共享常量。

#### P1 | 硬编码的配置值
| 配置项 | 值 | 位置 |
|--------|-----|------|
| 默认密码 | `"123456"` | `AdminService.java` 第 72/169 行 |
| VIP 每日成本 | `180` 经验值 | `UserService.java` 第 135 行 |
| 会话超时 | `24 小时`（日志却说 30 分钟） | `AppContextListener.java` 第 47/53 行 |

**建议**: 移入 `config.properties`。

#### P2 | 反规范化的计数字段缺乏对账机制
**表**: `article(likes)`, `ai_word_dic(likes/dislikes)`, `quiz_vocabulary(likes/dislikes)`, `quiz_comprehension(likes/dislikes)`

计数由应用层维护，无定期对账任务（`COUNT(*)` from vote 表 vs stored count），无监控。

**建议**: 实现定期对账作业，或仅实时计算而不存储（数据量小）。

#### P2 | `checkin()` 方法可能为死代码
**文件**: `UserService.java` — 定义了 `checkin` 方法但无任何 Servlet 调用。

**建议**: 若未使用则移除；若计划使用则添加对应端点。

#### P3 | 文章浏览历史用户本地存储，不在服务端记录
参照 `业务逻辑说明.md` 第 109 行 — 文章点赞仅在客户端 IndexedDB 去重，同一用户换设备/清缓存后可重复点赞，`article_like_count` 可能虚高。

**建议**: 在 `user_learned_article` 中增加 `liked` 字段或添加服务端投票表。

### 2.3 流程优化

#### P2 | 冷启动用户体验 — 词汇量测试入口过于突出
首次打开读物匹配页面弹出词汇量测试弹窗，未登录状态下也可以触发，但未提供"跳过"快捷入口。

**建议**: 在弹窗中增加"稍后再测"按钮，与"开始测试"并列展示。

#### P2 | 读者页加载失败无错误状态
**文件**: `frontend/src/pages/ReaderPage.vue` 第 153 行 — 文章加载时仅显示 spinner，若 API 失败则用户永久停留加载中。

**建议**: 添加"加载失败，点击重试"错误状态。

#### P3 | AI 缓存未设上限
**文件**: `frontend/src/stores/reader.js` 中 `aiExampleCache` 无内存上限（LRU/TTL/size cap），用户连续查词会无限增长。

**建议**: 设 LRU 缓存上限（例如 200 条）。

## 三、前端页面显示优化

### 3.1 组件化与复用

#### P1 | AdminPage.vue — 上传/编辑弹窗表单几乎一样（~180行重复）
**文件**: `frontend/src/pages/AdminPage.vue` 第 335-513 行

上传文章弹窗和修改文章弹窗共享相同的表单结构（标题、内容、来源、难度、AI 标题）。**建议**: 提取为 `ArticleFormModal` 组件，通过 `mode="create"|"edit"` prop 区分。

#### P1 | Toast/通知系统三份独立实现
| 文件 | 实现方式 |
|------|---------|
| `ProfilePage.vue` | `toastMsg` ref + `toastType` ref + `toastTimer` |

#### P1 | 分页逻辑两份完全一样的实现

5 页滑窗 + 首尾省略号 + 跳页输入框的逻辑完全一致，仅变量名不同。**建议**: 提取为 `<Pagination>` 共享组件，接收 `totalPages`/`currentPage`，emit `update:currentPage`。

#### P2 | 登录表单两份重复
**文件**: `LoginPage.vue` 和 `LoginModal.vue` — 表单字段/验证/提交逻辑 ~150 行重复。

**建议**: 提取为 `<LoginForm>` 组件。

#### P2 | 单词分词函数两份重复
**文件**: `ReaderPage.vue` `tokenizeText()` 和 `ArticleSidePanel.vue` `tokenizeText()` — 完全相同的正则 `/([a-zA-Z]+(?:['-][a-zA-Z]+)*)/g`。

**建议**: 提取到 `@/utils/tokenize.js`。

#### P2 | 头像文件列表两份重复
**文件**: `ProfilePage.vue` `avatarList` 和 `AppNav.vue` `avatarFiles`

**建议**: 提取到 `@/utils/avatars.js` 或 `@/constants/avatars.js`。

### 3.3 用户体验问题

#### P1 | 17+ 处使用原生 `alert()` 弹窗
**文件**: `AdminPage.vue` (8处), `ReaderPage.vue` (3处), `VocabTestModal.vue` (3处), `FirstVocabTestModal.vue` (2处), `ProfilePage.vue` (1+处)

原生 `alert()` 阻断式、不可定制样式。**建议**: 统一替换为 Toast 组件。

#### P1 | 全站无响应式布局
所有页面用固定列数 (`grid-cols-12` / `col-span-8` / `col-span-4`) 和固定宽度 (`max-w-[1200px]`, `w-[560px]`)，无 `lg:`/`md:` 断点。移动端视图不可用。

**建议**: 添加 `lg:` 断点前缀，移动端使用堆叠布局。

#### P1 | reader.js 中 ~300行 Mock 数据打包进生产
**建议**: 移除 mock 数据，或仅保留在 dev 模式。

#### P2 | 16 处 `console.log` 残余在生产构建
**文件**: `LoginModal.vue`, `historyDB.js`, `task.js`, `useAuth.js`, `useOnlineTimer.js`

**建议**: 移除或使用条件编译。

#### P2 | 无路由滚动行为
**文件**: `router/index.js` — 无 `scrollBehavior`，返回上一页时丢失滚动位置。

**建议**: 添加 `scrollBehavior: () => ({ top: 0 })`。

#### P2 | 无加载骨架屏
AdminPage 表格区域仅显示"加载中..."文字；MaterialsPage 有完整骨架屏实现可以作为参考。

**建议**: 统一添加 Skeleton 组件。

### 3.4 可访问性（Accessibility）

#### P2 | 全局缺失
- 无 `aria-label`/`aria-labelledby`/`role` 属性在交互元素上
- 模态框无焦点陷阱（Tab 可逃逸到背景元素）
- 模态框无 `@keydown.escape` 关闭
- 灰色文本 (`#9CA3AF`) 在白色背景对比度 ~2.9:1，低于 WCAG AA 4.5:1 标准

**建议**: 添加焦点管理、ESC 关闭处理，调暗文本色以满足对比度标准。

### 3.5 性能优化

#### P1 | ECharts 静态导入而非动态导入
**文件**: `ProfilePage.vue` — `import * as echarts from 'echarts'`（~1MB），仅用于趋势图和雷达图。

**建议**: 使用 `await import('echarts')` 动态导入。

#### P2 | `lemmatize.js` 中 400+ 行不规则动词词典
**建议**: 懒加载，仅在用户查词时加载。

#### P3 | ArticleSidePanel 和 ReaderPage 共享弹窗定位逻辑
**建议**: 提取为 `usePopoverPosition()` composable。

## 四、优化优先级路线图

### 第一阶段 — 安全修复（本周）
1. [P0] 移除 `/api/admin/` 的公开路径，从会话获取管理员身份
2. [P0] MD5 → bcrypt 密码哈希升级
3. [P0] 吊销并更换已泄露的 API Key 和数据库密码
4. [P0] 新增 `vip_expire_at` 列，修复 `last_checkin` 语义过载 Bug
5. [P0] 统一有效学习阶段定义
