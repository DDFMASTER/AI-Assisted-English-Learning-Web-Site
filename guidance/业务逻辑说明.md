# 英语学习网页软件 — 业务逻辑说明文档

---

## 1. 用户管理

### 1.1 注册与登录

- 所有未登录用户视为**游客**，可使用基础功能（如浏览文章、查词）。
- 注册时提供用户名（≤16字符，支持中英文/符号组合）、密码（加盐 MD5 加密）。
- 登录后系统维护会话，记录最近登录时间。

### 1.2 用户角色

| 角色           | 权限                              |
| ------------ | ------------------------------- |
| 游客（未登录）      | 基础查词、阅读文章、查看文章讲解                |
| 普通用户（normal） | 完整学习功能、记录学习进度、参与AI生成内容质量投票      |
| VIP用户（vip）   | 同普通用户，可手动生成全新的AI内容。             |
| 管理员（admin）   | 对词库、文章、题目对象进行增删改查，执行批量投票、软删除等操作 |

### 1.3 学习目的与水平

- 每位用户需选择当前学习阶段：**初中 / 高中 / 四级 / 六级 / 考研 / 托福**。
- 系统支持随时修改学习阶段，所有后续查词、文章推荐、题目生成均基于该阶段。
- 用户可进行**词汇量测试**，得分记录在 `literacy` 字段（合法得分为0~20000，预计大部分数据位于2000~8000），并记录最近测试日期。
- 词汇量与学习阶段的对应关系：初中(2500) / 高中(4500)  / 四级(5500) / 六级(6500) / 考研(7500) / 托福(10000)

### 1.4 签到与经验

- 每日签到功能：用户登录后可签到，更新 `last_checkin` 并增加 `experience` 值。用户每天只能签到一次。
- 经验值仅用于激励，暂不与等级或功能绑定，后续可扩展。

## 2. 词库与查词

### 2.1 六阶段静态词库

系统内置六张独立词库表，分别对应初中、高中、四级、六级、考研、托福。每张表存储该阶段单词、音标及释义。同一单词在同一阶段内可有多条记录（如 `miss` 有“错过”“思念”两义），每条记录有独立的流水 ID。

### 2.2 普通查词

- 用户点击文章中的单词时，系统根据当前学习阶段查询对应词库表。
- 定义：包含小于等于三个英文单词的内容为“单词”，包含大于三个英文单词的内容为“句子”。
- 若命中 **一条或多条** 结果，前端以翻页形式展示所有释义，并提示“该词有多种解释”。
- 若在该阶段词库中 **未命中**，前端提示“该词超出当前学习阶段”，同时向VIP用户提供 **“AI详解”** 按钮。
- 普通用户只能在阶段词库中查词，仅对VIP用户提供“AI详解”。

### 2.3 AI 详解

- VIP用户主动点击“AI详解”或在普通用户查词结果为 0 时发起 AI 查词。
- 后端先检查 `ai_word_dic`（AI 动态词典），查找该单词最新的未删除解释（`is_removed=0`），命中则直接返回。
- 若未命中，调用 AI 接口生成精讲内容（词义、例句、文化含义等），存入 `ai_word_dic` 并返回。
- 所有 AI 输入均严格从原文中提取，前端控制只能选取文段内容，以防止提示词注入。

### 2.4 词汇量测试

- 测试使用 `vocab_test` 表，内含 20000 个真实词（按使用频率排序）和 500 个伪词（形似真词的假词）。
- 测试按频率次序出题，用户判断是否为真词；伪词用于检测过度自信。
- 测试完成后得分记录到 `user.literacy`，作为后续个性化推荐的参考。

#### 2.4.1 词汇量测试原理

* **数据源** — COCA 20000 词频表，去重后约 17800 个独立词
* **抽样** — 9 个词频段分层抽样（101-500、501-1500、1501-3000、3001-5000、5001-7500、7501-10000、10001-13000、13001-16000、16001-20200），每段 7~13 个。
* **诱饵** — 约7~13个英语化但不存在的伪词（splanter、morbous、crentish 等）。最后总共抽取至少100词。
* **矫正** — 温和矫正：从每段认识率里扣掉 ½ × 诱饵命中率（pp），再外推到该段总词数

总词汇量 ≈ 100（默认掌握的功能词）+ Σ 各频段总词数 × max(0, 该段认识率 − ½×诱饵命中率)

置信区间用各频段二项分布方差汇总后取 ±1.96σ 给出 95% CI。

#### 2.4.2 词汇量测试在项目中的应用

用户测试获得的“总词汇量”会被填入`user - literacy`字段，同时根据用户的测试结果生成用户的学习阶段。

测试界面形式：以表格形式展示若干个单词，并设置勾选框。用户勾选“认识”的单词。

测试界面同时提供“更多单词”按钮，点击一次生成10个新的测试用单词（真实词和伪词比例同样在9:1附近浮动）（0:10, 1:9, 2:8, 3:7都是合法比例）

测试界面提示用户：“基础测试100个单词，不过测试得越多，结果越准。”

## 3. 文章系统

### 3.1 文章浏览

- `article` 表存储英语范文，每篇文章有标题、正文、来源、难度等级（与学习阶段一致）。
- 用户可按难度筛选文章，阅读过程中可点击单词查词（见 2.2），或划选句子/段落获取 AI 翻译（见下文 3.2）。

### 3.2 划词翻译（句子/段落）

- 用户划选文本后，前端根据**空格数量**判断：超过 3 个单词（即含 >2 个空格）视为句子或段落。
- 句子/段落翻译请求由 AI 处理，结果**仅缓存在用户本地 IndexedDB**，不经过服务端存储，不同设备或清除缓存后丢失。
- 该功能不涉及数据库表。

### 3.3 文章整体解读

- 每篇文章预留 `explanation` 字段（TEXT），初始为空。
- 用户首次请求“文章讲解”时，系统调用 AI 生成对该文章的整体解读（包含篇章大意、文化背景等），填充到 `explanation` 字段。
- VIP用户和普通用户都能请求生成文章讲解。
- 管理员可手动修改或重新生成该字段内容。
- 对解读内容，用户可**点赞 / 踩 / 取消**，系统通过 `ai_article_explain_vote` 表记录，并同步更新 `article` 表中的 `explanation_like_count` 和 `explanation_dislike_count`。

### 3.4 文章内容点赞

- 文章本身支持点赞，但**仅在用户本地存储点赞状态**（如 IndexedDB），不进行服务端去重。
- 这意味着同一用户更换设备或清除缓存后可再次点赞，`article_like_count` 仅作为展示参考，可能存在少许虚高。
- 该设计出于简化考虑，若未来需严格计数，可补充服务端投票记录表。

### 3.5 学习记录

- 用户完成一篇文章的阅读（及测验，见第 4 节）后，在 `user_learned_article` 中插入一条记录，关联用户 ID 与文章 ID，并记录完成时间。
- 该记录用于标记“已学文章”，避免重复推荐，并作为学习历史统计的基础。

## 4. 读后测验与题库

### 4.1 测验机制

- 用户首次阅读某篇文章（即 `user_learned_article` 中尚无记录）时，需要完成 **5 道词汇题 + 3 道阅读理解题**，全部作答后即视为学习完成。
- 用户答题的正确性由自己判断，系统**不自动批改**，即使用户故意全部选 A，也认为学习完成。
- 已经学习过的文章再次进入时，不再强制测验，但用户仍可手动调取题目进行练习。

### 4.2 题库管理

- **词汇题** 存储在 `quiz_vocabulary` 表，**阅读理解题** 存储在 `quiz_comprehension` 表。
- 每篇文章初始题目数为 0。当第一名用户请求该文章题目时，AI 一次性生成 5 道词汇题和 3 道阅读理解题，插入对应表，并更新 `article` 表的 `vocquiz_num` 和 `comquiz_num`。
- 后续用户请求题目时，若该文章当前**词汇题少于 50 道**或**阅读理解题少于 30 道**，系统先调用 AI **额外生成 1 道题** 并入库，然后从所有未删除的题目中随机抽取所需数量。
- 为避免用户抽到重复题目，前端缓存已答题目的 ID（存于 IndexedDB），每次抽取时排除这些 ID；若题库不足，系统优先返回所有可用题目。
- 题目支持 **软删除**：当某题的“不喜欢”数超过可配置阈值（如 30）或管理员手动操作时，标记 `is_removed=1`，不再参与抽取。被软删除的题目由定时任务定期物理清理。

### 4.3 题目评价

- 用户可对每一道词汇题或阅读理解题进行 **赞 / 踩 / 取消** 操作。
- 投票记录在 `ai_vocquiz_vote` 和 `ai_comquiz_vote` 表中，联合主键保证一人一题一条记录，`action` 取值 0/1/2 分别代表赞、踩、取消。
- 点赞/踩的总数通过事务同步更新题目表中的 `like_count` 和 `dislike_count`。

## 5. 管理员功能

### 5.1 内容管理

- 管理员可对词库、文章、AI 讲解、题目等进行增删改查操作。
- 可手动修改文章整体解读、AI 词典中的精讲内容，或触发重新生成。

### 5.2 批量操作与审计

- 管理员可通过特殊工具执行批量投票（如一次性为一篇文章讲解增加 100 个赞），操作详情记录在 `admin_action_log` 表。
- 日志包含操作对象类型、目标 ID、操作类型、参数和时间戳，便于审计和回溯。
- 管理员的所有敏感操作均需通过白名单校验，防止非法操作记录。

## 6. 安全与缓存策略

### 6.1 防提示词注入

- 由前端设计保证，所有可发送给 AI 的文本均节选自数据库中的文章原文或单词表，不接受用户自由输入的文本。
- 前端仅传递起始位置或选中文本。

### 6.2 缓存策略

- **服务端**：不再使用独立的 AI 缓存表，改为“动态生成 + 随机抽取”模式。AI 讲解和题目一经生成即持久化，后续用户复用，总量随使用逐步增长。
- **客户端**：
  - 单词精讲内容、文章段落翻译结果等，存储在浏览器 IndexedDB 中，同设备再次访问时直接读取，避免重复请求。
  - 已答题目 ID 列表同样本地存储，用于去重抽题。
  - 文章点赞状态仅存本地，不跨设备同步。

## 7. 业务流程图（概要）

```
用户登录/游客访问
   │
   ├─→ 选择学习阶段（或修改）
   ├─→ 词汇量测试（可选，记录得分）
   ├─→ 阅读文章
   │     ├─ 点击单词 → 阶段词库查词（多条时翻页展示）→ 无则提示并可选AI详解 → 查询/生成AI精讲
   │     └─ 划选句子/段落 → 前端判断类型 → AI翻译（结果仅本地缓存）
   ├─→ 文章讲解（首次触发AI生成，可赞/踩）
   ├─→ 完成学习 → 记录到已学文章
   │     └─ 首次学习 → 强制测验（5词+3阅）→ 题目动态生成与抽取
   └─→ 管理员：管理内容、批量操作（日志记录）
```
