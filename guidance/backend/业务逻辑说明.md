# 英语学习网页软件 — 业务逻辑说明文档

---

## 1. 用户管理

### 1.1 注册与登录

- 所有未登录用户视为**游客**，可浏览文章列表、查看单词释义等基础功能。
- 注册时提供用户名（≤16字符，支持中英文/符号组合）、密码（加盐 MD5 加密）。
- 登录后系统维护会话，记录最近登录时间。

### 1.2 用户角色

| 角色           | 权限                              |
| ------------ | ------------------------------- |
| 游客（未登录）      | 基础查词、阅读文章、查看文章讲解                |
| 普通用户（normal） | 完整学习功能、记录学习进度、参与AI生成内容质量投票      |
| VIP用户（vip）   | 同普通用户，可手动生成全新的AI内容。             |
| 管理员（admin）   | 对词库、文章、题目对象进行增删改查，执行批量投票、软删除等操作 |

### 1.3 学习目的与水平

- 每位用户需选择当前学习阶段：**初中 / 高中 / 四级 / 六级 / 考研 / 托福**。
- 系统支持随时修改学习阶段，所有后续查词、文章推荐、题目生成均基于该阶段。
- 用户可进行**词汇量测试**，得分记录在 `literacy` 字段（合法得分为0~20000，预计大部分数据位于2000~8000），并记录最近测试日期。
- 词汇量与学习阶段的对应关系：初中(2500) / 高中(4500)  / 四级(5500) / 六级(6500) / 考研(7500) / 托福(10000)

### 1.4 签到与经验

- 每日签到功能：用户登录后可签到，更新 `last_checkin` 并增加 `experience` 值。用户每天只能签到一次。
- 经验值仅用于激励，暂不与等级或功能绑定，后续可扩展。

## 2. 词库与查词

### 2.1 六阶段静态词库

系统内置六张独立词库表，分别对应初中、高中、四级、六级、考研、托福。每张表存储该阶段单词、音标及释义。同一单词在同一阶段内可有多条记录（如 `miss` 有“错过”“思念”两义），每条记录有独立的流水 ID。

### 2.2 普通查词

- 用户点击文章中的单词时，系统根据当前学习阶段查询对应词库表。
- 定义：包含小于等于三个英文单词的内容为“单词”，包含大于三个英文单词的内容为“句子”。
- 若命中 **一条或多条** 结果，前端以翻页形式展示所有释义，并提示“该词有多种解释”。
- 若在该阶段词库中 **未命中**，前端提示”该词超出当前学习阶段”，同时向用户提供 **”AI详解”** 按钮。
- 普通用户在阶段词库查词结果为 0 时可触发 AI 详解；VIP 用户除此之外还可手动主动调用 AI 详解（即使在词库中已命中）。

### 2.3 AI 详解

- VIP 用户可随时手动点击”AI详解”按钮发起 AI 查词；普通用户在阶段词库查词结果为 0 时自动获得 AI 详解入口。
- 后端先检查 `ai_word_dic`（AI 动态词典），查找该单词最新的未删除解释（`is_removed=0`），命中则直接返回。
- 若未命中，调用 AI 接口生成精讲内容（词义、例句、文化含义等），存入 `ai_word_dic` 并返回。
- 所有 AI 输入均严格从原文中提取，前端控制只能选取文段内容，以防止提示词注入。

### 2.4 词汇量测试

系统提供两种词汇量测试模式，均使用 `vocab_test` 表（内含 20000 个真实词按使用频率排序和 500 个伪词），测试完成后得分记录到 `user.literacy`。两个测试接口均返回防缓存响应头 `Cache-Control: no-cache, no-store, must-revalidate`。

#### 2.4.1 传统模式（VocabTestService）
- **接口**：`GET /api/vocabtest/words`
- 测试按 COCA 频率次序出题，用户以勾选框（check/uncheck）形式判断是否认识单词；伪词用于检测过度自信。
- 测试界面以表格形式展示单词，用户勾选"认识"的单词；提供"更多单词"按钮，点击一次生成 10 个新测试用单词（真实词和伪词比例在 9:1 附近浮动，0:10、1:9、2:8、3:7 均为合法比例）。
- 测试界面提示用户："基础测试 100 个单词，不过测试得越多，结果越准。"

#### 2.4.2 卡片模式（VocabTestCardService）
- **接口**：`GET /api/vocabtest/cards`
- 从 `vocab_test` 表 `ORDER BY RAND()` 随机抽取单词，以选择题形式呈现。
- 选项由 AI 生成（含正确释义和干扰项），后端使用静态 `ConcurrentHashMap` 选项缓存（`optionCache`）按单词缓存已生成的选项，避免重复调用 AI。
- 前端以卡片翻页形式展示，每张卡片一个单词配四个选项，用户选择后即时反馈正误。

#### 2.4.3 词汇量测试原理

* **数据源** — COCA 20000 词频表，去重后约 17800 个独立词
* **抽样** — 9 个词频段分层抽样（101-500、501-1500、1501-3000、3001-5000、5001-7500、7501-10000、10001-13000、13001-16000、16001-20200），每段 7~13 个。
* **诱饵** — 约7~13个英语化但不存在的伪词（splanter、morbous、crentish 等）。最后总共抽取至少100词。
* **矫正** — 温和矫正：从每段认识率里扣掉 ½ × 诱饵命中率（pp），再外推到该段总词数

总词汇量 ≈ 100（默认掌握的功能词）+ Σ 各频段总词数 × max(0, 该段认识率 − ½×诱饵命中率)

置信区间用各频段二项分布方差汇总后取 ±1.96σ 给出 95% CI。

#### 2.4.4 词汇量测试在项目中的应用

用户测试获得的”总词汇量”会被填入 `user.literacy` 字段，同时根据用户的测试结果生成用户的学习阶段。

前端提供”快速测试”（传统勾选模式）和”精确测试”（卡片选择题模式）两种入口，用户可在个人中心选择测试方式。

## 3. 文章系统

### 3.1 文章浏览

- `article` 表存储英语范文，每篇文章有标题、正文、来源、难度等级（与学习阶段一致）。
- 用户可按难度筛选文章，阅读过程中可点击单词查词（见 2.2），或划选句子/段落获取 AI 翻译（见下文 3.2）。

### 3.2 划词翻译（句子/段落）

- 用户划选文本后，前端根据**空格数量**判断：超过 3 个单词（即含 >2 个空格）视为句子或段落。
- 句子/段落翻译请求由 AI 处理，结果**仅缓存在用户本地 IndexedDB**，不经过服务端存储，不同设备或清除缓存后丢失。
- 该功能不涉及数据库表。

### 3.3 文章整体解读

- 每篇文章预留 `explanation` 字段（TEXT），初始为空。
- 用户首次请求“文章讲解”时，系统调用 AI 生成对该文章的整体解读（包含篇章大意、文化背景等），填充到 `explanation` 字段。
- VIP用户和普通用户都能请求生成文章讲解。
- 管理员可手动修改或重新生成该字段内容。
- 对解读内容，用户可**点赞 / 踩 / 取消**，系统通过 `ai_article_explain_vote` 表记录，并同步更新 `article` 表中的 `explanation_like_count` 和 `explanation_dislike_count`。

### 3.4 文章内容点赞

- 文章本身支持点赞，后端提供 `POST /api/article/like` 接口，接收 `articleId` 和 `action`（`like`/`unlike`）参数，直接对 `article` 表的 `article_like_count` 字段进行增减。
- 取消点赞时使用 `GREATEST(article_like_count - 1, 0)` 防止计数为负。
- 前端在本地 localStorage 中按文章 ID 存储点赞状态（key 格式：`aael_liked_<articleId>`），用于展示当前用户是否已点赞，但不作为服务端去重的强制依据。
- 点赞按钮（爱心图标）位于 ReaderPage 导航栏中书签按钮旁边，红色表示已点赞，hover 显示 tooltip 和当前点赞数。

### 3.5 学习记录

- 用户将文章滚动至底部（阅读进度达到 100%），系统自动判定为阅读完成，在 `user_learned_article` 中插入一条记录，关联用户 ID 与文章 ID，并记录完成时间。
- 该记录用于标记”已学文章”，避免重复推荐，并作为学习历史统计的基础。
- 测验不再是完成阅读的强制条件，而是作为可选的辅助练习工具（见第 4 节）。

## 4. 阅读理解出题（可选辅助练习）

### 4.1 当前机制

- 阅读完成判定：用户将文章滚动至底部（阅读进度 100%）即视为完成阅读，**不再强制要求完成测验**。
- 阅读理解题目作为**可选的辅助练习工具**，用户在阅读器侧边栏可自主调出。题目由 AI 基于文章内容实时生成，通过 `/api/article/quiz` 接口返回 JSON，**不再持久化到数据库**。
- VIP 用户和非 VIP 用户均可使用阅读理解出题功能。非 VIP 用户每日限制 3 篇文章的 AI 出题服务。

### 4.2 历史题库（已废弃）

> ⚠️ **以下内容已过时，仅作为历史记录保留。**

- `quiz_vocabulary`（词汇题库）和 `quiz_comprehension`（阅读理解题库）两张表已废弃，作为冗余表保留在数据库中，业务代码不再读写这两张表。
- `article` 表中的 `vocquiz_num` 和 `comquiz_num` 字段不再维护。
- 对应的投票表 `ai_vocquiz_vote` 和 `ai_comquiz_vote` 也随之一并废弃。
- 原设计中的题目持久化、动态补充、软删除、投票评价等机制均已停用。当前 AI 出题采用”实时生成、用完即弃”的模式，无需长期存储。

## 5. 管理员功能

### 5.1 内容管理

- 管理员可对词库、文章、AI 讲解、题目等进行增删改查操作。
- 可手动修改文章整体解读、AI 词典中的精讲内容，或触发重新生成。

### 5.2 批量操作与审计

- 管理员可通过特殊工具执行批量投票（如一次性为一篇文章讲解增加 100 个赞），操作详情记录在 `admin_action_log` 表。
- 日志包含操作对象类型、目标 ID、操作类型、参数和时间戳，便于审计和回溯。
- 管理员的所有敏感操作均需通过白名单校验，防止非法操作记录。

## 6. 安全与缓存策略

### 6.1 防提示词注入

- 由前端设计保证，所有可发送给 AI 的文本均节选自数据库中的文章原文或单词表，不接受用户自由输入的文本。
- 前端仅传递起始位置或选中文本。

### 6.2 缓存策略

- **服务端**：不再使用独立的 AI 缓存表，改为”动态生成 + 随机抽取”模式。AI 讲解一经生成即持久化，后续用户复用，总量随使用逐步增长。阅读理解题目为实时生成、用完即弃（详见 §4.1）。
- **词汇测试选项缓存**：`VocabTestCardService` 使用静态 `ConcurrentHashMap`（`optionCache`）按单词缓存 AI 生成的选项，避免对同一单词重复调用 AI 接口。
- **客户端**：
  - 单词精讲内容、文章段落翻译结果等，存储在浏览器 IndexedDB 中，同设备再次访问时直接读取，避免重复请求。
  - 已答题目 ID 列表同样本地存储，用于去重抽题。
  - 文章点赞状态仅存本地，不跨设备同步。

## 7. 业务流程图（概要）

```
用户登录/游客访问
   │
   ├─→ 选择学习阶段（或修改）
   ├─→ 词汇量测试（可选，记录得分）
   ├─→ 阅读文章
   │     ├─ 点击单词 → 阶段词库查词（多条时翻页展示）→ 无则提示并可选AI详解 → 查询/生成AI精讲
   │     ├─ 划选句子/段落 → 前端判断类型 → AI翻译（结果仅本地缓存）
   │     └─ 可选：阅读理解出题（AI实时生成，不持久化）
   ├─→ 文章讲解（首次触发AI生成，可赞/踩）
   ├─→ 滚动至底部 → 自动记录完成 → 写入 user_learned_article
   └─→ 管理员：管理内容、批量操作（日志记录）
```

## 8. 阅读时间估算

`ReadingTimeUtil.calculate(contentLength, articleDifficulty, studyPurpose)` 为文章卡片提供预计阅读时间，便于用户规划学习节奏。

**计算逻辑**：
- 将文章字符数除以 5 估算英文单词数（`contentLength / 5`）。
- 根据用户当前学习阶段确定基准 WPM（每分钟阅读词数）：初中 80 / 高中 100 / 四级 120 / 六级 140 / 考研 160 / 托福 180。
- 根据文章难度与用户水平的差距调整速度：每差一个等级调整 ±15%。例如四级用户阅读托福难度文章，WPM 降低 15% × 2 = 30%。
- 最终结果四舍五入到 0.5 分钟，最低 0.5 分钟。

前端展示格式为 "X min read" 或 "X.5 min read"。

## 9. 登录安全

### 9.1 登录限流（LoginAttemptService）

- 基于内存的登录尝试限流器，按用户名/IP 维度记录失败尝试次数。
- **阈值**：10 次连续失败登录后触发锁定，锁定时长 15 分钟。
- **自动清理**：超过 30 分钟无活动的记录自动清除，防止内存泄漏。
- 登录失败时，服务端在错误响应中返回 `remainingAttempts`（剩余尝试次数）和 `lockSeconds`（锁定剩余秒数），前端可据此提示用户。

## 10. 文章点赞 API

### 10.1 接口流程

- **接口**：`POST /api/article/like`
- **请求参数**：`articleId`（文章 ID）、`action`（`like` 或 `unlike`）
- **服务端逻辑**：直接更新 `article` 表的 `article_like_count` 字段 —— `like` 时 +1，`unlike` 时使用 `GREATEST(article_like_count - 1, 0)` 防止负数。
- **前端状态**：localStorage 中以 `aael_liked_<articleId>` 键存储当前文章是否已点赞，用于按钮状态展示（红色实心/空心爱心）。
- **按钮位置**：ReaderPage 导航栏，位于书签按钮旁边，hover 显示 tooltip 和当前点赞数。
