# EngliAI — 在线英语学习平台项目总结

---

## 一、项目背景

在全球化和信息化深度融合的时代背景下，英语作为国际通用语言，已经深入到教育、科研、商务、文化交流等各个领域。对于中国学生而言，英语阅读能力和词汇积累是语言学习的核心环节，然而传统英语教学模式面临诸多挑战：教材内容更新缓慢、学习资源获取不便、缺乏个性化学习路径、课后练习形式单一、学生难以获得即时反馈等。

本项目 **EngliAI（AI-Assisted English Learning）** 正是为解决上述问题而设计开发的一款在线英语学习平台。项目面向**教育行业**和**外语学习领域**，以网站形式提供快速便捷的访问和学习途径，用户无需安装任何客户端软件，通过浏览器即可随时随地开展英语学习。

平台深度融合了人工智能技术，借助大语言模型（DeepSeek API）为用户提供优质阅读素材和具有挑战性的练习题目。核心价值体现在以下几个方面：

1. **个性化阅读推荐**：根据用户的英语水平等级（CEFR）和学习阶段，智能匹配合适难度的英语文章，确保阅读材料既具有挑战性又不至于过于困难，使用户始终处于最佳学习区。

2. **AI 辅助深度阅读**：阅读过程中提供单词释义查询、段落翻译、文化背景讲解、AI 例句生成等功能，帮助用户深入理解文章内容，扩展文化视野，而非停留在浅层的单词记忆层面。

3. **科学的词汇量评估**：采用基于 COCA 词频表的分层抽样算法和伪词诱饵机制，能够较为准确地评估用户的词汇量水平，为个性化学习提供数据支撑。

4. **自适应测评体系**：基于 CEFR（欧洲语言共同参考框架）标准，通过 AI 动态生成测评题目并评估用户的多维度能力（词汇、语法、阅读、文化、逻辑），帮助用户清晰认知自身英语水平并及时调整学习策略。

5. **游戏化激励机制**：通过每日任务、经验值（XP）获取、VIP 兑换等机制，激发用户的学习动力，将枯燥的语言学习转变为富有成就感的持续过程。

本项目可有效辅助学生提高英语阅读能力和词汇量，适用于中学英语课外阅读、大学四六级备考、考研英语训练、托福备考等多种学习场景。同时，平台的管理后台为教师或管理员提供了内容管理和用户监控的便捷手段，使平台能够灵活适应不同教学管理需求。

## 二、项目内容

EngliAI 平台围绕"通过阅读提升词汇量和理解能力"这一核心目标，构建了从用户水平评估、个性化内容推荐、沉浸式阅读学习、读后练习巩固到学习趋势追踪的完整学习闭环。各功能模块之间层次分明、紧密关联。

### 2.1 用户系统与水平评估

#### 2.1.1 注册与登录

首次访问平台的用户为**游客**身份，可浏览文章列表但不能使用交互功能。用户需注册账户并登录后才能完整体验平台功能。注册时需填写用户名（支持中英文，最长 16 字符）、密码（采用加盐 MD5 加密存储），并选择当前学习阶段（初中 / 高中 / 四级 / 六级 / 考研 / 托福）。系统还提供了默认密码账户创建功能，方便管理员批量导入用户。

#### 2.1.2 首次词汇量快速测试

用户首次注册后，系统自动弹出**词汇量快速测试**弹窗（FirstVocabTestModal）。该测试以交互式卡片形式展示一系列英语单词，用户通过点击"认识"或"不认识"按钮作答。测试采用基于 COCA 20000 词频表的分层抽样算法，从 9 个词频段各抽取 7-13 个真实单词，同时混入约 7-13 个形似真词的伪词诱饵。通过伪词命中率矫正过度自信偏差，最终估算用户的词汇量，并将结果（词汇量数值及对应 CEFR 等级）与后续文章推荐和测评系统绑定。

#### 2.1.3 用户角色体系

系统设有三种用户角色：

| 角色 | 说明 |
|------|------|
| **普通用户（normal）** | 可阅读文章、查词、做测验、使用基本 AI 功能（有每日次数限制） |
| **VIP 用户（vip）** | 除普通用户功能外，AI 功能无次数限制，可主动调用 AI 生成新内容 |
| **管理员（admin）** | 拥有独立管理后台，可管理用户、文章，查看系统日志 |

普通用户可通过完成每日任务获取经验值（XP），累计足够的 XP 后可在个人中心兑换 VIP 身份（180 XP/天）。

### 2.2 智能读物匹配与阅读器

#### 2.2.1 读物匹配页

登录后进入**读物匹配页**（MaterialsPage），系统根据用户的 CEFR 水平等级自动筛选对应难度的文章列表。每篇文章以卡片形式展示，包含标题、来源、难度标签、点赞数等信息。用户可以按难度等级手动筛选文章。

#### 2.2.2 AI 沉浸式阅读器

点击文章进入**阅读器页面**（ReaderPage），这是平台的核心学习界面。阅读器提供以下交互功能：

- **单词查询**：点击文章中的任意单词，系统根据用户当前学习阶段查询对应词库表（初中/高中/四级/六级/考研/托福共六张静态词库），以弹窗展示单词的音标和释义。若当前阶段词库未命中，提示用户该词超出当前阶段，VIP 用户可进一步使用 AI 详解功能。

- **AI 单词详解**：当词库未命中或 VIP 用户主动请求时，后端先查询 AI 动态词典（ai_word_dic 表），未命中则调用 DeepSeek API 生成精讲内容（含音标、释义、详细解释），存入词典并返回。

- **AI 例句生成**：点击单词弹窗中的"生成例句"按钮，AI 为该单词生成 3 个包含中文翻译的例句，帮助用户在上下文中理解单词用法。

- **段落翻译**：用户划选文章中的句子或段落（超过 3 个单词），系统自动判断为段落类型，调用 AI 进行中英文翻译。翻译结果缓存在浏览器 IndexedDB 中，同设备重复访问时直接读取。

- **文化背景讲解**：阅读器侧边栏提供"文化背景分析"功能，AI 分析文章中的文化知识点（如节日、习俗、历史事件等），以结构化卡片形式展示，每张卡片包含标题和中英文内容。

- **阅读理解出题**：AI 基于文章内容生成阅读理解选择题，包含题目、选项、正确答案和解析。题目持久化到数据库，后续用户可复用。

- **文章整体讲解**：AI 对文章进行整体解读，概括篇章大意，分析文章结构和写作特点。用户可对讲解内容点赞/踩/取消。

### 2.3 读后测验与学习记录

#### 2.3.1 测验机制

用户首次阅读某篇文章时，需要完成 **5 道词汇题 + 3 道阅读理解题**。题目由 AI 生成并持久化到数据库（quiz_vocabulary 表和 quiz_comprehension 表）。每题包含 4 个选项和正确答案。用户答题后系统记录学习完成状态，已学文章不再强制测验，但用户可随时手动调取题目进行练习。

#### 2.3.2 题目管理与去重

每篇文章的词汇题上限为 50 道，阅读理解题上限为 30 道。当首次生成的题目不够或用户数量增长后，系统自动调用 AI 额外生成题目入库。前端通过 IndexedDB 缓存已答题目的 ID，每次请求时排除已答题目，避免用户抽到重复题。

#### 2.3.3 学习记录

用户完成文章阅读和测验后，系统在 `user_learned_article` 表中记录学习时间，构成用户学习历史的基础数据，同时避免重复推荐已学文章。

### 2.4 自适应测评中心

**测评中心**（AssessmentPage）是用户检验学习成果、获取进阶反馈的核心模块：

1. **首题即时返回**：用户开始测评后，AI 生成首道题目并立即返回（约 15-20 秒），让用户快速进入答题状态。

2. **后台异步出题**：剩余 9 道题目由 AI 在后台异步生成，前端通过轮询机制每 3 秒拉取新题目，生成一道展示一道。

3. **难度自适应**：出题难度基于用户当前 CEFR 等级（从词汇量测试结果推算），确保题目难度与用户水平匹配。若用户水平提升，下一轮测评的题目难度也会相应调整。

4. **答案去偏**：为防止连续 3 题答案相同影响测评效度，系统会在 AI 出题时注入约束条件，避免答案分布不均。

5. **AI 多维评估**：提交全部答案后，AI 从词汇（vocabulary）、语法（grammar）、阅读（reading）、文化（culture）、逻辑（logic）五个维度对用户表现进行评估，给出综合得分和 CEFR 等级。

6. **CEFR 晋级机制**：单次测评最高增加 20% 进度（满分 100 分即 `/5 = 20%`），分数低于 30 分不增加进度。进度达到 100% 时自动晋升一个 CEFR 等级（如 B1 → B2）。测评结果以**雷达图**和详细报告形式在结果页（ResultPage）展示。

### 2.5 个人中心

**个人中心**（ProfilePage）汇聚用户的学习数据和个性化设置：

- **词汇量详细测试**：用户可随时在个人中心进行更详细的词汇量测试。该测试同样采用分层抽样算法，AI 生成释义选项和伪词诱饵，提供比首次快速测试更准确的词汇量评估结果。支持"更多单词"功能，每次额外生成 10 个测试词，测试越多结果越精确。

- **学习趋势**：以图表形式展示用户的每日在线学习时长记录，支持设置目标学习时间。系统通过会话监听器和 IndexedDB 自动追踪用户的在线时长。

- **学习记录**：展示用户已完成的文章列表和完成时间，方便回顾学习历程。

- **生词本**：用户在阅读过程中查询的单词自动加入生词本（存储于 IndexedDB），最大容量 200 词，支持随时查看和复习。

- **错题集**：用户在测验和测评中答错的题目自动记录，支持回顾错题和正确答案。

- **设置**：可修改目标学习时间、当前学习阶段（不影响英语水平，但影响词书查询逻辑）。支持自定义头像，内置多种头像选项。

- **经验值与 VIP 兑换**：展示当前 XP 余额，支持使用 XP 兑换 VIP 天数（180 XP/天）。

### 2.6 每日任务与签到

- **每日签到**：用户每天可签到一次，获得 10 点经验值。
- **每日任务**：系统设置每日学习任务（如完成一篇文章阅读），完成后可获取额外的经验值奖励。

### 2.7 管理后台

**管理后台**（AdminPage）仅供 admin 角色用户访问，提供以下功能：

- **用户管理**：查询所有用户列表（含用户名、角色、学习阶段、词汇量、经验值、注册时间等），支持删除普通用户（级联清理投票记录和学习记录），支持重置用户密码为默认密码。

- **在线用户监控**：查看当前在线用户列表，支持对普通用户执行**强制下线**操作（通过 kickedUserIds 集合实现）。

- **文章管理**：支持增删查改文章（标题、正文、来源、难度等级），删除文章时级联清理关联的词汇题、阅读理解题、学习记录和投票数据。

- **IP 请求日志**：通过请求日志监听器（RequestLogListener）记录所有 API 请求的 IP 地址、请求路径、耗时等信息，管理员可在后台查询和分析。

### 2.8 功能关联总结

各功能模块之间形成了紧密的学习闭环：

```
注册/登录 → 词汇量快速测试 → 读物匹配 → 沉浸式阅读
    ↓                                         ↓
个人中心 ← 测评反馈 ← 自适应测评 ← 完成读后测验
    ↓
经验值/VIP ← 每日签到与任务
```

用户从水平评估开始，经历个性化推荐、深度阅读、练习巩固、自适应测评、趋势追踪的完整流程。AI 功能贯穿始终——查词、翻译、文化讲解、例句生成、出题、评估——各 AI 功能共享同一 DeepSeek API 基础设施，通过不同的 prompt 模板和解析器实现差异化的功能输出。

## 三、技术路线

### 3.1 开发环境与工具

### 3.2 软件架构

本项目基于经典的 **MVC（Model-View-Controller）** 架构模式进行设计，并采用改进的 **JSP Model2** 模型，实现了**前后端分离**的开发模式。整体架构分为三个层次：

```
┌─────────────────────────────────────────────────┐
│                   客户层 (Client)                 │
│         Vue 3 SPA / 浏览器 IndexedDB             │
│    (Vue Router + Pinia + Tailwind CSS)           │
└─────────────────┬───────────────────────────────┘
                  │ HTTP/REST API (JSON)
┌─────────────────▼───────────────────────────────┐
│                 Web 层 (Controller)              │
│              Nginx (反向代理)                     │
│              Tomcat 10 (Servlet 容器)             │
│         Servlet (控制器, 接收请求)               │
│         Filter (登录过滤器, 权限校验)            │
│         Listener (会话监听, 请求日志)            │
└─────────────────┬───────────────────────────────┘
                  │
┌─────────────────▼───────────────────────────────┐
│               业务逻辑层 (Service)                │
│    UserService / ArticleService / WordService    │
│    AIService / AdminService / VocabTestService   │
│    MonitorService / HealthService                │
└─────────────────┬───────────────────────────────┘
                  │
┌─────────────────▼───────────────────────────────┐
│              数据访问层 (DAO)                     │
│    UserDAO / ArticleDAO / WordDAO                 │
│    AdminActionLogDAO                              │
└─────────────────┬───────────────────────────────┘
                  │
┌─────────────────▼───────────────────────────────┐
│              数据层 (Model)                       │
│    MySQL 数据库 + IndexedDB (浏览器端)            │
│    Entities (数据实体对象)                        │
└─────────────────────────────────────────────────┘
```

#### 3.2.1 Web 层（Servlet + Filter + Listener）

Web 层是 MVC 架构中的 **Controller**，负责接收前端 HTTP 请求、进行权限校验、调用业务逻辑层处理数据、返回 JSON 响应。

- **Servlet 控制器**：平台共包含 30+ 个 Servlet 类，每个 Servlet 负责一组相关的 API 端点。如 `UserLoginServlet` 处理登录请求、`ArticleDetailServlet` 返回文章详情、`AssessmentGenerateServlet` 触发 AI 出题等。

- **LoginFilter（登录过滤器）**：拦截所有非公开 API 请求，验证用户身份。从请求头或 Session 中提取用户 token，校验通过后将请求转发给目标 Servlet，未通过则返回 401。

- **SessionListener（会话监听器）**：监听 HTTP 会话的创建和销毁事件，维护在线用户计数（通过 AtomicInteger）和用户-会话 ID 映射（通过 ConcurrentHashMap），为管理员后台提供在线用户列表数据。

- **RequestLogListener（请求日志监听器）**：记录每个 HTTP 请求的 IP 地址、请求方法、请求路径、响应状态码和耗时，存储到应用上下文的线程安全列表中，供管理员查询分析。

- **AppContextListener（应用上下文监听器）**：在应用启动时初始化全局属性（在线计数器、用户会话映射、被踢用户集合、启动时间戳），在应用关闭时清理资源。

#### 3.2.2 业务逻辑层（Service）

业务逻辑层是 MVC 架构中的核心处理层，封装了所有业务规则和流程控制：

- **UserService**：处理用户注册、登录、签到、经验值累加、VIP 兑换、VIP 过期检测等核心用户业务流程。密码使用加盐 MD5 哈希存储，VIP 到期后自动降级。

- **AIService**：封装所有 AI 相关功能的调用逻辑，统一通过 DeepSeek API 进行通信。核心职责包括：

  - **单词例句生成**：调用 DeepSeek，传入 system prompt（`word-examples.txt`）和单词，要求返回结构化 JSON（含 en/zh 字段对）。
  - **段落翻译**：传入翻译专用 prompt 和段落文本，解析返回的 translation 字段。
  - **文化背景分析**：传入文章全文和 culture-analysis prompt，解析 title/content/zh 三元组。
  - **阅读理解出题**：传入文章和 quiz-generation prompt，解析 question/optionA-D/answer/explanation 字段。
  - **测评出题**：分单题快速版本和批量生成版本，根据学习阶段（studyPurpose）生成适配难度的短文章和选择题。
  - **测评评估**：传入题目及正确答案 + 用户答案，AI 返回多维度评分和 CEFR 等级。
  - **AI 标题生成**：为无标题文章自动生成英文标题。
  - **AI 查词**：当所有词书未命中时，由 AI 搜索单词释义，结果存入 AI 动态词典。

  所有 AI 调用的流程统一：
  1. 构建 system prompt + user content
  2. 通过 Java 内置 `java.net.http.HttpClient` 发送 POST 请求到 DeepSeek API
  3. 接收 JSON 响应，提取 choices[0].message.content
  4. 规范化 JSON（去除 markdown 代码块包裹、压缩空白字符）、解析字段
  5. 封装为结构化结果对象返回

- **VocabTestService**：词汇量测试的核心算法服务。加载 COCA 20000 词频数据，将单词按频率分为 9 个频段（101-500、501-1500、...、16001-20000），同时加载 500 个伪词。采用分层随机抽样算法，从每个频段抽取 7-13 个真实词，混合 7-13 个伪词，共约 100 个测试词。估算算法：对各频段计算矫正认识率 r_i = max(0, p_i_raw − 0.5 × pf)，再将矫正率外推到频段总词数求和，加 100 个默认功能词，得出总词汇量估计值；同时使用二项分布方差汇总计算 95% 置信区间。

- **AdminService**：管理员业务逻辑，包含创建/删除用户、重置密码、增删改文章等操作。删除用户和文章时使用数据库事务保证级联清理的原子性。所有管理员关键操作记录到 `admin_action_log` 审计日志表。

- **ArticleService / WordService / MonitorService / HealthService / VocabTestCardService**：分别负责文章推荐与匹配、词库查询与维护、系统监控、健康检查、词汇量测试卡生成等业务。

#### 3.2.3 数据访问层（DAO）

- **DBUtil**：数据库连接工具类，管理数据库连接池和事务支持。
- **UserDAOImpl / ArticleDAOImpl / WordDAOImpl / AdminActionLogDAOImpl**：分别封装对用户表、文章表、词库表（六张）、审计日志表的增删改查操作。

- **MySQL 数据库**：使用 MySQL 关系型数据库存储用户信息、文章内容、词库数据、题目数据、投票记录、学习记录、AI 动态词典、审计日志等持久化数据。

### 3.4 服务端部署架构

项目在阿里云 ECS 云服务器上完成部署，部署架构如下：

```
用户浏览器
    │
    │ DNS 解析 (阿里云 DNS)
    ▼
域名 (HTTP/HTTPS)
    │
    ▼
Nginx (反向代理服务器)
    │ 根据域名和路径规则分发请求
    │
    ├─→ 静态资源 (前端 SPA 文件)
    │   Tomcat webapps/ 目录
    │
    └─→ API 请求 (/api/*)
        Tomcat HTTP Connector (端口 8080)
            │
            ▼
        Servlet 容器 (Jakarta Servlet 5.0)
```

- **阿里云 DNS**：配置域名解析，将用户域名指向 ECS 服务器的公网 IP 地址。
- **Nginx**：作为反向代理服务器，监听 80/443 端口，截获域名请求。静态资源（前端 HTML/CSS/JS）直接由 Nginx 返回或转发到 Tomcat 的静态资源路径；API 请求通过 `proxy_pass` 指令反向代理到 Tomcat 的 8080 端口。Nginx 还负责 SSL 终端、Gzip 压缩和请求限流等。
- **Tomcat 10**：作为 Servlet 容器和 Web 服务器，部署 WAR 包（由 Maven 构建）。Tomcat 10 使用 Jakarta Servlet 5.0 规范（`jakarta.servlet.*` 命名空间），与 JDK 21 兼容。
- **前后端统一部署**：前端 Vue 项目构建后的静态文件放入 Tomcat 的 `webapps/` 目录，与后端 Servlet 由同一个 Tomcat 实例提供服务，避免了跨域问题。

### 3.5 算法说明

#### 3.5.1 词汇量分层抽样估计算法

**原理**：基于 COCA（Corpus of Contemporary American English）20000 词频表，将英语单词按使用频率从高到低排序，分为 9 个频段。每个频段的总词数已知（如 101-500 名含 400 词）。测试时从每个频段随机抽取 7-13 个样本单词，混入约 7-13 个伪词（英语化但实际不存在的单词），用户逐个判断是否认识。

**估算公式**：

$$Vocabulary \approx 100 + \sum_{i=1}^{9} N_i \times \max\left(0, \frac{k_i}{n_i} - \frac{1}{2} \times \frac{k_f}{n_f}\right)$$

其中 $N_i$ 为第 i 频段的总词数，$k_i/n_i$ 为该频段的原始认识率，$k_f/n_f$ 为伪词命中率（代表用户过度自信程度）。95% 置信区间基于各频段二项分布方差汇总后取 ±1.96σ 计算。

**AI 辅助增强**：AI 生成词汇测试选项时，为每个真实单词生成 4 个中文释义选项（含 1 个正确释义和 3 个干扰项），为伪词生成相似但不存在的释义，进一步提高测试准确性。

#### 3.5.2 CEFR 等级映射算法

用户词汇量测试结果与 CEFR 等级的映射关系：

| CEFR 等级 | 对应词汇量 | 中文标签 |
|-----------|-----------|---------|
| A1 | 500 | 初级 |
| A2 | 1500 | 初级上 |
| B1 | 3000 | 中级 |
| B2 | 5000 | 中高级 |
| C1 | 8000 | 高级 |
| C2 | 12000 | 精通 |

#### 3.5.3 伪词矫正机制

在词汇量测试中，用户对伪词的误判率用于矫正对真实词的过度自信。矫正系数为 0.5，即从各频段原始认识率中扣除伪词命中率的一半，避免高估。

#### 3.5.4 渐进式题目生成与轮询

测评模块采用"首题即时返回 + 后台异步生成 + 前端轮询拉取"的策略：AI 首先快速生成首道题目（约 15-20 秒）让用户立即进入答题状态，剩余 9 道题在后台异步生成，前端每 3 秒轮询一次拉取新题。这种设计将用户感知的等待时间压缩到首题生成时间，大幅提升体验。
