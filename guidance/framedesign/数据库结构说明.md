> **补充说明**：项目还维护了两类**非数据库存储结构**：
> - **请求日志**（`RequestLogEntry`）：以 `List` 形式存储于 ServletContext 内存中（上限 10000 条），通过 `MonitorService` 定时持久化为 JSON 文件（每日轮转、过期自动清理）。
> - **PK 游戏房间**（`GameRoom`）：以 `ConcurrentHashMap<String, GameRoom>` 存储于 ServletContext（key = 4 位房间码），纯内存数据。WAITING 房间超过 30 分钟自动清理；FINISHED 房间保留 5 分钟（供双方查看结果）后自动清理。应用重启后全部清空。

### 3.2 词库模块

#### 六张学习阶段词库表（`junior`, `senior`, `cet_4`, `cet_6`, `postgraduate`, `toefl`）

六张表结构完全一致，以下以 `junior` 为例：

| 字段            | 类型           | 约束                         | 说明      |
| ------------- | ------------ | -------------------------- | ------- |
| `id`          | BIGINT       | PK, AUTO_INCREMENT         | 流水主键    |
| `word`        | VARCHAR(64)  | NOT NULL, INDEX `idx_word` | 单词      |
| `phonetic`    | VARCHAR(64)  | DEFAULT NULL               | 英式音标    |
| `translation` | VARCHAR(255) | DEFAULT NULL               | 该阶段单词释义 |

**索引与约束**：

- PRIMARY KEY (`id`)
- INDEX `idx_word` (`word`)

> **设计要点**：  
> 同一单词在同一阶段内可能存在多条记录（不同释义），因此不再将 `word` 设为主键或唯一键，而是使用自增 `id` 作为主键，并对 `word` 建立普通索引以加速查询。

#### `vocab_test` 词汇量测试表（静态数据）

| 字段          | 类型          | 约束                     | 说明                                       |
| ----------- | ----------- | ---------------------- | ---------------------------------------- |
| `vocab_id`  | INT         | PK                     | 词汇频率次序 ID（真实词 1-20000，伪词 >20000）         |
| `word`      | VARCHAR(64) | NOT NULL               | 测试单词（同一单词可能因大小写/词形变体出现多次，故不设唯一约束）        |
| `is_pseudo` | TINYINT(1)  | NOT NULL, DEFAULT 0    | 是否伪词（0=真词，1=伪词）                          |

**索引与约束**：

- PRIMARY KEY (`vocab_id`)

> **设计要点**：  
> 词汇量测试数据来源于 COCA 20000 词频表（约 17800 个去重词）以及约 500 个伪词。该表为静态只读表，不应执行 UPDATE/DELETE 操作。
> 卡片式测试（`VocabTestCardService`）通过 `ORDER BY RAND()` 每次随机抽取 90 真词 + 10 伪词，确保不同测试间单词不重复。

#### `ai_word_dic` AI 动态词典（词语精讲缓存）

| 字段              | 类型           | 约束                                  | 说明                     |
| --------------- | ------------ | ----------------------------------- | ---------------------- |
| `aidic_id`      | BIGINT       | PK, AUTO_INCREMENT                  | 词典条目 ID                |
| `word`          | VARCHAR(64)  | NOT NULL                            | 单词拼写                   |
| `phonetic`      | VARCHAR(64)  | DEFAULT NULL                        | AI 生成的音标               |
| `translation`   | VARCHAR(255) | DEFAULT NULL                        | AI 生成的简短释义             |
| `explanation`   | TEXT         | DEFAULT NULL                        | AI 词语精讲内容（词义、例句、文化含义等） |
| `like_count`    | INT          | NOT NULL, DEFAULT 0                 | 点赞数                    |
| `dislike_count` | INT          | NOT NULL, DEFAULT 0                 | 不喜欢数                   |
| `is_removed`    | TINYINT(1)   | NOT NULL, DEFAULT 0                 | 软删除标记（1=已删除）           |
| `created_at`    | DATETIME     | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间                   |

**索引与约束**：

- PRIMARY KEY (`aidic_id`)
- INDEX `idx_word_active` (`word`, `is_removed`, `created_at` DESC)  
  *用于快速获取某单词最新的有效解释记录*

## 四、核心业务逻辑对应关系

1. **单词查询**  
   
   - 优先查当前学习阶段对应的词库表（如 `junior`），通过 `WHERE word = ?` 返回所有释义（可能多条）。  
   - 若无结果或用户主动要求 AI 详解，则查询 `ai_word_dic`（取 `is_removed=0` 的最新记录）；仍无结果时调用 AI 生成并插入该表。

2. **AI 动态词典维护**  
   
   - `ai_word_dic` 中的每一条解释独立存在，用户可点赞/踩，通过 `ai_word_vote` 记录并更新冗余计数。  
   - 当 `dislike_count` 超过阈值或管理员操作时，设置 `is_removed=1` 进行软删除。

3. **文章学习与讲解**  
   
   - 用户完成阅读后，在 `user_learned_article` 中记录。  
   - 首次请求文章整体讲解时，AI 生成内容并填入 `article.explanation`。  
   - 讲解的点赞/踩通过 `ai_article_explain_vote` 记录，同步更新 `explanation_like_count` 与 `explanation_dislike_count`。

4. **题目动态填充（已废弃/待扩展）**  
   
   - 原设计：每篇文章初始 `vocquiz_num=0`，`comquiz_num=0`，用户首次请求题目时 AI 生成题目入库，后续动态补充和随机抽取。  
   - **现状：** 此机制已于 2026-07 废弃。阅读理解题目改为 AI 实时生成，通过 `/api/article/quiz` 接口直接返回 JSON，不再持久化到 `quiz_vocabulary` 和 `quiz_comprehension` 表。两张表及对应的投票表作为冗余数据保留。  
   - **扩展方向：** 这些表可重新激活用于题目持久化缓存、用户答题历史追踪、社区出题等场景（详见 `未来扩展空间.md`）。

5. **用户投票机制**  
   
   - 四张投票表的 `action` 字段三态（0/1/2），联合主键保证一人一内容一条记录。  
   - 应用层通过事务同步更新相应内容表的 `like_count` / `dislike_count`。

6. **管理员操作审计**  
   
   - 所有批量操作（如批量点赞、批量删除）均记录至 `admin_action_log`，便于追溯。

7. **词汇量测试**  
   
   - 新版卡片测试（`VocabTestCardService` / `/api/vocabtest/cards`）从 `vocab_test` 表按 `ORDER BY RAND()` 随机抽取 90 真词 + 10 伪词，AI 生成释义选项。  
   - 旧版勾选测试（`VocabTestService` / `/api/vocabtest/words`）基于 COCA 词频文件和伪词文件（`vocabtest/COCA.txt`、`vocabtest/fakewords.txt`）分层抽样，不依赖 `vocab_test` 表。

## 五、注意事项

- 所有与 AI 交互的输入内容均在后端校验，防止提示词注入。
- 投票表等高频写入表建议定期归档或清理无效数据。
- 未使用外键约束，数据一致性由应用层事务保证。
- `vocab_test` 为静态只读表，不应执行 UPDATE/DELETE 操作。
- 词库表的 `id` 仅作为物理主键，业务查询仍依赖 `word` 索引。
- 请求日志（`RequestLogEntry`）和 PK 房间（`GameRoom`）为内存级数据结构，非数据库表。日志通过 JSON 文件实现重启后恢复；房间通过 `cleanupExpired()` 自动清理（WAITING 30 分钟，FINISHED 5 分钟），应用重启后全部清空。
